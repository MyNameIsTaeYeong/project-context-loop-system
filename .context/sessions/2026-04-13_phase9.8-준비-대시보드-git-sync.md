### 2. 웹 대시보드에서 Git 코드 문서 조회 지원 (커밋 1)

**문제**: code_summary/code_doc/git_code 문서가 DB에 있지만 대시보드에서 필터링/조회 불가.

**구현:**
- `dashboard.html`: Source 필터에 Code Doc / Code Summary / Git Code 옵션 추가
- `app.css`: source_type별 배지 색상 — code_doc(초록), code_summary(파랑), git_code(보라)
- `document_list.html`: 배지 클래스 동적 적용 (`badge-source-{{ doc.source_type }}`)
- `tab_original.html`: code_doc/code_summary → 마크다운 렌더링(marked.js, Rendered/Raw 토글), git_code → 구문 강조(highlight.js, 확장자 기반 언어 감지)
- `document_detail.html`: code_doc/code_summary/git_code일 때 Sources 탭 표시
- **신규 `tab_sources.html`**: document_sources 양방향 연결 표시 (Source Code Files / Referenced By)
- `documents.py`: `tab_sources` 라우트 + `_guess_language()` 헬퍼 (확장자→언어 매핑 30+개)
- `metadata_store.py`: `get_stats()`에 git 소스 타입별 카운트 추가
- `document_stats.html`: Git 문서 있을 때 2번째 통계 카드 행 표시

### 3. Level 1 파일별 요약 DB 저장 (커밋 2)

**문제**: FileSummary(Level 1)는 파이프라인 실행 중 메모리에만 존재하고 DB에 저장되지 않음.

**구현:**
- `coordinator.py`: `store_file_summary()` 메서드 추가
  - source_type: `code_file_summary`, source_id: `{product}:{relative_path}`
  - content_hash 기반 멱등성 (동일 내용 재저장 시 스킵)
  - document_sources: code_file_summary → git_code 자동 연결
- `run_and_store()`: Level 2 저장 전에 Level 1 파일별 저장 루프 추가
- 대시보드: File Summary 필터(노란색 배지), 마크다운 렌더링, Sources 탭

**문서 계층 완성:**
| 타입 | 배지 | 설명 |
|------|:----:|------|
| code_file_summary | 노란색 | Level 1 — 파일별 LLM 요약 |
| code_summary | 파란색 | Level 2 — 디렉토리 종합 문서 |
| code_doc | 초록색 | Level 3 — 카테고리별 관점 문서 |
| git_code | 보라색 | 원본 소스 코드 |

### 4. Git Sync 페이지 (커밋 3)

**목적**: 웹 대시보드에서 Git 레포 동기화를 시작/모니터링.

**구현:**
- **신규 `api/git_sync.py`**: 백엔드 API + 인메모리 상태 관리
  - `POST /api/git-sync/start`: CoordinatorAgent.run_and_store() 백그라운드 실행
  - `GET /api/git-sync/status`: 실행 상태 JSON (idle/running/completed/failed)
  - 동시 실행 방지 (asyncio.Lock, 이미 실행 중이면 409)
  - SyncStatus dataclass로 상태/단계/경과시간/결과/오류 추적
- **신규 `git_sync.html`**: 3섹션 구성
  - Configuration: 레포/상품/카테고리/LLM 엔드포인트 설정 표시 (읽기 전용) + validate() 경고
  - Sync Control: Start/Running/Completed/Failed 4개 상태별 UI
  - Generated Documents: Git 관련 문서 최신 50개 목록
- **신규 `git_sync_status.html`**: HTMX 3초 폴링 파셜
- `base.html`: 네비게이션에 "Git Sync" 링크 추가
- `app.py`: git_sync_router 등록

### 5. LLM 클라이언트 생성 버그 수정 (커밋 4)

**문제**: `_run_sync()`에서 `LLMWorkerAgent(git_config)` 호출 — 생성자는 `(worker_llm, synthesizer_llm)` 2개의 LLMClient를 받음.

**수정:**
```python
# Before (잘못됨)
worker = LLMWorkerAgent(git_config)
category_agent = LLMCategoryAgent(git_config)

# After (수정됨)
worker_llm = git_config.build_llm_client("worker")
synthesizer_llm = git_config.build_llm_client("synthesizer")
worker = LLMWorkerAgent(worker_llm, synthesizer_llm)

orchestrator_llm = git_config.build_llm_client("orchestrator")
category_agent = LLMCategoryAgent(orchestrator_llm)
```

## 설계 검토 — 미구현 (논의만)

### 3단계 문서 생성의 컨텍스트 효율성

RAG 관점에서 Level 1/2/3 각각의 가치를 분석:
- **Level 1 (파일 요약)**: 필수. 벡터 검색에 가장 적합한 검색 단위. 정밀하고 원본 추적 가능.
- **Level 2 (디렉토리 종합)**: 부분적 가치. "모듈 전체 구조" 질의에는 유용하나 벡터 인덱싱에서는 노이즈 가능. 대시보드 열람 + 그래프 추출 소스로 역할 변경 검토.
- **Level 3 (카테고리 관점)**: 가치 의문. 비용 높고, 관점이 사전 고정됨 (RAG에서는 질의가 관점을 결정해야 함). 같은 정보가 3번 저장되는 중복 문제. 제거 권장.
- **결론**: Phase 9.8 연결 후 실제 검색 품질 측정해서 최종 결정 예정.

### Git Sync 실패 복구 (resume-from-failure)

현재 문제: LLM 결과가 인메모리에만 존재 → Stage 5-6 DB 저장 실패 시 Stage 3-4 LLM 결과 전체 소멸.

제안 (방안 A+B 조합):
- **방안 A**: `run_and_store()` 즉시 저장으로 리팩토링 — Worker 디렉토리 완료 즉시 DB 저장
- **방안 B**: 재실행 시 이미 처리된 디렉토리 스킵 옵션
- **방안 C (Phase 9.9)**: sync_history 테이블 + commit_hash 추적 (증분 처리와 함께 구현)
