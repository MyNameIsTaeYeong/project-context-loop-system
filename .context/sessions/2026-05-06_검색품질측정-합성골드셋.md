## 핵심 진단 — git_code 멀티뷰 임베딩 비대칭 (P0)

`pipeline.py:128-179` (git_code) vs `181-273` (confluence/upload) 비교 결과:

| 항목 | git_code | confluence/upload |
|---|---|---|
| 임베딩 입력 | `title + parent + name + signature + docstring` 만 | **멀티뷰**: body 본문 + meta(title+section_path) |
| 본문 임베딩 | **없음** | 있음 (`#body`) |
| 경로/메타 임베딩 | **없음** | 있음 (`#meta`) |

`_clamp_max_per_tenant` 같은 추상적 함수명 안에 "rate limit" 도메인 용어가
들어 있어도, 임베딩 벡터에는 시그니처+독스트링만 들어가서 자연어 질의가
본문 의미와 매칭되지 않는다. 한국어 질의는 거의 항상 본문 의미에
가깝게 묻는데 인덱싱은 영어 식별자만 보고 있는 셈.

**해결 방향 (별도 PR 필요)**: `git_code` 분기에도 Confluence 와 동일한
멀티뷰 (`#body` + `#meta`) 적용. `_search_chunks` 의 `logical_chunk_id`
dedup 이미 양쪽을 흡수하므로 검색 단계 변경 불필요.

## 핵심 설계

### 1. 합성 골드셋 인프라 (`592fa92`)

`src/context_loop/eval/` 신규 패키지:

- `metrics.py` — 결정론적 순수 함수: Recall@k / Precision@k / MRR /
  nDCG@k / hit@k / aggregate. 검색 결과 ID 시퀀스 + 정답 ID 집합 형태로
  통일.
- `gold_set.py` — `GoldItem` / `GoldSet` 데이터 모델 + YAML 직렬화.
  외부 의존 없이 pyyaml 만 사용.
- `synth.py` — LLM 합성 + 3단계 품질 게이트:
  1. 답변 가능성 (Judge LLM, 출처 청크로 답할 수 있는가)
  2. 식별자 누출 (결정론 — 청크 고유 식별자가 질문에 그대로 들어갔는가)
  3. 일반성 (Judge LLM, 무관 청크로도 답할 수 있는가)

스크립트 2개:
- `scripts/build_synthetic_gold_set.py` — 인덱싱된 청크 → 합성 골드셋
- `scripts/eval_search.py` — 골드셋 → CSV per-query + JSON summary

**역방향 생성 패턴**: 청크를 보여주고 "이 청크가 정답이 되도록 사람이
물어볼 만한 질문 N개" 를 LLM 에 요청. 라벨링 비용 0.

**식별자 누출 방지 핵심 트릭**: 프롬프트에 "식별자 그대로 베끼지 말고
의미 단위로 풀어쓰라" 명시. 이게 없으면 BM25 만으로도 100% 맞히는
사기성 골드셋이 됨.

### 2. `web/app._build_llm_client` 재사용 (`fe2c6d9`)

기존 스크립트는 `EndpointLLMClient` 를 직접 빌드하여 `llm.provider` 분기와
`llm.headers` (사내 LLM 게이트웨이 헤더), `llm.reasoning_profiles`
(Qwen3 `enable_thinking` 등) 가 단편적으로만 전달됨. `mcp/server.py` 와
동일 패턴으로 정리.

`src/context_loop/eval/llm.py:build_llm_client(config, **overrides)`:
- override 가 모두 비면 `_build_llm_client(config)` 그대로 호출 →
  provider 분기 + headers + reasoning_profiles 자동 적용
- override 가 있으면 `config.llm.*` 키를 try/finally 로 임시 덮어쓰고
  빌더 호출 후 복원 — Generator/Judge 두 번 호출해도 격리 보장

### 3. `config.eval.{role}.*` 운영 디폴트화 (`c2ee695`, `7e0c14d`)

기존 `sources.git.processing.{worker,synthesizer,orchestrator}` 와 동일
패턴으로, 평가용 LLM 설정을 config 로 이전. CLI 인자는 일회성 override.

```yaml
eval:
  generator:                       # 골드셋 질의 합성
    endpoint: ""
    model: ""
    api_key: ""
    headers: {}
    reasoning_profiles: {}         # 모델 family 별 다름 — role 단위 정의 가능
  judge:                           # 답변 가능성·품질 채점
    endpoint: ""
    model: ""
    api_key: ""
    headers: {}
    reasoning_profiles: {}
```

우선순위: CLI override > `config.eval.{role}.*` > `config.llm.*` 폴백.

`build_eval_llm_client(config, role)` + `role_is_configured()` 헬퍼.
시스템 = Qwen3, judge = DeepSeek 같은 family 분리 시나리오에서 reasoning
페이로드 형식 차이 (`enable_thinking` vs `thinking`) 를 role 단위로
별도 매핑 가능.

### 4. 합성 파라미터 config 화 (`86cb560`) → 사용자 요청으로 revert (`9d68dba`)
