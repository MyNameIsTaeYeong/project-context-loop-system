# 2026-04-29 /api/chat 성능 최적화 — 병렬화 + 전용 리랭커 도입

## 배경

사용자 보고: "현재 /api/chat 의 응답이 느립니다. 아마 LLM 호출 때문에 그런 것 같은데
LLM 호출 중 병렬화로 시간을 줄일 수 있는 부분이 있는지 검토해주세요."

`/api/chat` 의 RAG 파이프라인을 추적해 LLM 호출 4건을 식별:
1. HyDE 가상 문서 생성 (`hyde_enabled` 시)
2. 그래프 탐색 계획 (`include_graph` 시)
3. **LLM 기반 리랭킹 (`rerank_enabled` 시)** ← 이번 세션 주 타겟
4. 최종 답변 생성

이 중 ②와 ③이 직렬로 실행 중이었지만 데이터 의존성 분석 결과 서로 무관함을
발견 — 병렬화 가능. 이어 사용자 요청으로 ③의 LLM 호출 자체를 dedicated
cross-encoder 리랭커 모델 호출로 교체.

## 핵심 설계 결정

### 1. 리랭킹 ↔ 그래프 탐색 병렬화 (D-045 부분)

**진단**: `assemble_context_with_sources` 가
1. 벡터 검색 → 2. rerank LLM 호출 → 3. graph plan LLM 호출 → 4. 최종 답변 LLM
순서로 직렬 실행. ②는 `chunk_results` 만, ③은 `query_embedding` 만 의존.

**해결**: `asyncio.gather(_maybe_rerank(), _maybe_graph())` 로 두 LLM 호출을
동시 실행. LLM RTT 한 번 분량의 지연 단축.

**구조**: 동일 패턴이 `assemble_context` (MCP) 에도 필요 → 헬퍼
`_rerank_and_search_graph(...)` 모듈 내부 함수로 추출. 두 호출 경로의 동기화
누락 위험 제거.

### 2. 전용 리랭커 모델 클라이언트 도입 (D-045 부분)

**계기**: 사용자 요청 — "리랭커용 전용 모델을 사용하려 합니다. LLM 호출을
리랭커용 모델로 대체해주세요. 호출 시 body 는 model, query, documents 를
보내고 있어요."

**결정**: dedicated cross-encoder 리랭커 API (Cohere/Jina/TEI 호환)와 통신하는
독립 클라이언트 도입.

- `processor/reranker_client.py` 신규: `RerankerClient` ABC +
  `EndpointRerankerClient`.
  - 요청 본문: `{"model": ..., "query": ..., "documents": [...]}`
  - 응답 파서 `parse_rerank_response()` 가 4가지 형식 자동 정규화:
    - Cohere/Jina: `{"results": [{"index", "relevance_score"}, ...]}`
    - TEI: `[{"index", "score"}, ...]`
    - 단순 dict: `{"scores": [float, ...]}`
    - 단순 list: `[float, ...]`
- `processor/reranker.py` 단순화: LLM 프롬프트, JSON 추출, 정규식 폴백 모두
  제거. `reranker_client.rerank(query, documents)` 한 번 호출 후 점수 매핑.
  130 → 80 라인.
- 점수 의미 변경: 기존 0~10 (LLM scoring) → 0~1 (모델별 상이, 보통
  cross-encoder cosine/logit). `reranker_score_threshold` 기본값 4.0 → 0.3.
- config 신규 섹션 `reranker:` (기존 `llm:` 와 동일 구조 — endpoint/model/
  api_key/headers).
- `rerank_enabled=true` + `reranker_client=None` 이면 리랭킹 단계 자동 스킵
  (graceful — `reranker.endpoint` 미설정 시 `_build_reranker_client` 가 None
  반환).

### 3. include_source_code 파라미터 누락 수정

**진단**: `assemble_context` / `assemble_context_with_sources` 에는 매개변수가
존재했지만 호출자(`/api/chat`, MCP `search_context`) 에서 전달되지 않아
**항상 False** 로 동작. code_doc/code_summary 의 원본 git 소스 첨부 기능이
사실상 죽어 있었음.

**수정**:
- `ChatRequest` 에 `include_source_code: bool = True` 필드 추가.
- MCP `search_context` 도구에 `include_source_code: bool = True` 파라미터 추가.
- 두 호출 사이트에서 `assemble_context*` 로 전달.
- 기본값 True (사용자 요청) — 코드 문서가 핵심 use case 라 기본 첨부가 자연스러움.

### 4. 응답 max_tokens 상향

`/api/chat` 답변 생성의 `max_tokens` 2048 → 8192. 답변이 길어질 때 잘리는
증상 방지.

## 트러블슈팅 — `assemble_context` 에 누락된 병렬화 적용

PR #2 직후 사용자 질문: "해당 변경이 MCP 호출 시에도 적용되는 건가요?"

확인: `/api/chat` 은 `assemble_context_with_sources` 사용, MCP 툴은
`assemble_context` 사용 — 첫 커밋(`5e9f3d9`)은 후자만 수정해 MCP 경로는
여전히 직렬이었음. 같은 검색 파이프라인이 두 함수에 중복 구현되어 있어
한쪽만 변경 시 누락 위험이 구조적으로 존재.

해결: `_rerank_and_search_graph` 헬퍼를 모듈 레벨로 추출(`98a8c47`) 하고 두
호출자 모두 동일 헬퍼를 사용하도록 정리. 향후 검색 파이프라인 변경 시 한
곳만 수정하면 됨.

## 미해결 — 차기 작업

### 우선순위 1: HyDE 단계 임베딩-LLM 병렬화 (이번 세션 미적용)
초기 검토에서 식별한 추가 병렬화 지점 — `expand_query_embedding` 내부에서
"원본 쿼리 임베딩" 과 "HyDE 가상 문서 LLM 생성" 이 직렬. `asyncio.gather`
로 묶을 수 있음. HyDE 활성화 시에만 효과가 있어 우선순위는 낮음. 사용자가
HyDE 를 켜는 시점에 처리.

### 우선순위 2: `/api/chat` 스트리밍 응답
최종 답변 LLM 호출의 첫 토큰 지연(TTFB) 을 줄이려면 병렬화가 아닌 스트리밍
응답 전환이 필요. `LLMClient.complete` 자체는 이미 내부적으로 stream=True 로
동작하지만 응답을 전부 받아 한 번에 반환 중. SSE 기반 스트리밍 엔드포인트는
별도 작업.
