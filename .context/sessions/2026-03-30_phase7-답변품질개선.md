# 2026-03-30~04-01 세션: RAG 파이프라인 답변 품질 개선 (Phase 7.1~7.6)

## 배경

- 전체 RAG 파이프라인(Confluence 입수 → 청킹/그래프화 → 검색 → 답변 생성)의 답변 품질 분석 수행

## 작업 내역

### Phase 7.1: HTML→Markdown 변환기 개선 (I-012)

**신규 파일**: `src/context_loop/ingestion/html_converter.py`

- **변경 전**: `confluence.py`의 `_html_to_markdown()`이 정규식 기반으로 HTML→MD 변환. 테이블(`<table>`) 태그 strip, Confluence 매크로(`<ac:structured-macro>`) 무시, 중첩 목록 flat 변환.
- **변경 후**: BeautifulSoup 전처리 + markdownify 변환으로 교체. 공유 모듈 `html_converter.py`를 신규 생성하여 `confluence.py`와 `mcp_confluence.py` 양쪽에서 사용.
- **Confluence 매크로 전처리 지원**:
  - info/warning/note/tip 패널 → `<blockquote>` → 마크다운 blockquote
  - code/noformat 매크로 → `<pre><code>` → 펜스드 코드 블록
  - expand 매크로 → 본문 펼침
  - toc/children/recently-updated 매크로 → 제거
  - `ac:image` → `<img>`, `ac:link` → `<a>` 태그 변환
- **영향 범위**:
  - `confluence.py`: `_html_to_markdown()` → `html_converter.html_to_markdown()` 위임
  - `mcp_confluence.py`: `convert_html_to_markdown()` → 동일 모듈로 위임, `markdownify` 직접 import 제거
  - `pyproject.toml`: `beautifulsoup4>=4.12.0` 의존성 추가
- **테스트**: 28개 신규 테스트 (`tests/test_ingestion/test_html_converter.py`)
- **설계 결정**: D-018

### Phase 7.2: 헤딩 기반 계층적 청킹 + 섹션 메타데이터 (I-013)

**파일**: `src/context_loop/processor/chunker.py`

- **변경 전**: 텍스트를 `\n\n`(빈 줄) 기준으로 단락 분리 후 토큰 기반 청킹. 헤딩 구조 무시. 메타데이터에 `chunk_index`와 `title`만 저장.
- **변경 후**: 마크다운 헤딩(`#`~`######`)을 인식하여 섹션별로 분리한 뒤 각 섹션 내에서 토큰 기반 청킹. 각 청크에 `section_path` (상위 헤딩 경로) 첨부.
- **주요 변경점**:
  - `Chunk` dataclass에 `section_path: str` 필드 추가
  - `_split_into_sections()`: 헤딩 스택 기반 계층 경로 계산 (예: `"프로젝트 > 아키텍처 > 백엔드"`)
  - 헤딩 텍스트를 청크 본문에 포함하여 임베딩 시 검색 정확도 향상
  - 헤딩 없는 문서는 기존과 동일하게 단락 기반 분할 (하위 호환)
- **영향 범위**:
  - `pipeline.py`: 벡터스토어 메타데이터에 `section_path` 포함
  - `context_assembler.py`: 검색 결과 포맷팅 시 섹션 경로 표시 (`_format_chunk_results`, `assemble_context_with_sources`)
- **테스트**: 8개 신규 테스트, 기존 포함 197개 통과
- **설계 결정**: D-019

### Phase 7.3: Cross-encoder Reranker + 유사도 Threshold (I-015)

**신규 파일**: `src/context_loop/processor/reranker.py`

- **변경 전**: 벡터 검색이 top-K 결과를 유사도 무관하게 반환. `"API Gateway 포트"` 질의에 대해 관련 없는 `"회사 소개"` 청크도 포함. 검색 결과가 단순 concat으로 LLM에 전달.
- **변경 후**: 두 단계 필터링 파이프라인 도입.
  1. **유사도 threshold**: cosine similarity < 0.3인 무관한 청크를 사전 제외
  2. **LLM 리랭킹**: 남은 청크를 LLM에 한꺼번에 전달하여 0~10점 관련도 평가 → 점수순 재정렬 → 점수 threshold 미만 추가 제외
- **주요 변경점**:
  - `reranker.py`: `rerank()` — 단일 LLM 호출로 전체 청크 평가 (비용 최소화). LLM 실패 시 원본 순서 유지 (graceful degradation).
  - `context_assembler.py`: `_search_chunks()`에 `similarity_threshold` 파라미터 추가. `assemble_context()`, `assemble_context_with_sources()`에 리랭킹 파라미터 4개 추가.
  - `tools.py`: MCP 도구에서 config 기반 설정 전달.
  - `chat.py`: 웹 채팅 API에서 config 기반 설정 전달.
  - `default.yaml`: `search` 섹션 추가 (threshold, reranker 설정).
- **테스트**: 16개 신규 테스트 (리랭커 12개 + context_assembler 4개), 기존 포함 213개 통과
- **설계 결정**: D-020

## 주요 개선 효과 (Phase 7.3)

### 무관한 청크 제거 (threshold)
- **Before**: `"API Gateway 포트"` 질의 → top-10에 유사도 0.15인 `"회사 소개"` 포함
- **After**: `similarity_threshold: 0.3` → 유사도 0.3 미만 청크 자동 제외, LLM 컨텍스트 노이즈 감소

### 정밀한 관련도 재평가 (reranker)
- **Before**: bi-encoder 유사도 순위만으로 결과 정렬. 키워드 매칭은 높지만 의미적으로 무관한 청크가 상위에 위치 가능
- **After**: LLM이 질의-청크 쌍의 의미적 관련도를 0~10으로 평가. `"API Gateway 설정 방법"` 질의에 대해 설정 방법이 포함된 청크가 단순 언급 청크보다 상위로 재정렬

### Phase 7.4: 그래프 추출 전체 문서 처리 — Map-reduce (I-014)

**파일**: `src/context_loop/processor/graph_extractor.py`

- **변경 전**: `content[:4000]`으로 앞 4000자만 LLM에 전달. 10,000자 문서에서 후반 60%의 엔티티/관계 완전 누락. Confluence 문서의 핵심 아키텍처 정보가 목차/배경 뒤에 있으면 추출 불가.
- **변경 후**: 문서 길이에 따라 자동 분기 — 4000자 이하는 기존과 동일한 단일 호출, 초과 시 map-reduce:
  1. **Map**: 문서를 `max_content_chars` 크기로 분할 (단락 → 줄바꿈 → 강제 순서로 자연 경계 존중), 각 청크에서 독립적으로 그래프 추출
  2. **Reduce**: 전체 결과 병합 — `(name.lower(), entity_type)` 기준 엔티티 중복 제거, `(source.lower(), target.lower(), relation_type)` 기준 관계 중복 제거
- **주요 변경점**:
  - `extract_graph()`: 길이 판단 후 `_extract_single()` 또는 `_extract_map_reduce()` 분기
  - `_split_content()`: 단락/줄바꿈 경계 존중 분할
  - `_merge_graphs()`: 대소문자 무시 중복 제거 + 빈 설명 보충
  - `_parse_graph_response()`: 공통 파싱 로직 추출 (중복 코드 제거)
  - `_USER_PROMPT_CHUNK_TEMPLATE`: 섹션 `(N/M)` 정보 포함
  - 부분 실패 허용: 한 청크 실패 시 나머지만 병합
  - `pipeline.py` 변경 없음 (함수 시그니처 동일)
- **테스트**: 14개 신규 테스트 (split 5 + merge 6 + map-reduce 통합 3), 기존 포함 229개 통과
- **설계 결정**: D-021

### Phase 7.5: 쿼리 확장 — HyDE (Hypothetical Document Embedding) (I-016)

**신규 파일**: `src/context_loop/processor/query_expander.py`

- **변경 전**: 사용자 쿼리("배포 절차")가 그대로 임베딩되어 벡터 검색. 문서에 "릴리즈 프로세스", "CI/CD 파이프라인" 등 다른 용어를 사용하면 어휘적 갭으로 검색 누락.
- **변경 후**: HyDE 활성화 시 LLM이 질의에 대한 가상 답변 문서를 생성 → 원본 쿼리 임베딩과 가상 문서 임베딩을 평균하여 의미적으로 풍부한 검색 벡터 생성.
- **주요 변경점**:
  - `query_expander.py`: `generate_hypothetical_document()` — 3~5문장 가상 답변 (temperature=0.7). `expand_query_embedding()` — 원본+HyDE 평균 벡터. 실패 시 원본 임베딩 반환.
  - `context_assembler.py`: 두 함수에 `hyde_enabled` 파라미터 추가. 활성화 시 `expand_query_embedding()` 사용.
  - `tools.py`, `chat.py`: config에서 `search.hyde_enabled` 전달.
  - `default.yaml`: `search.hyde_enabled: false` 추가.
- **테스트**: 11개 신규 (query_expander 9 + context_assembler 2), 기존 포함 240개 통과
- **설계 결정**: D-022

## 주요 개선 효과 (Phase 7.5)

### 동의어/용어 갭 해소 (HyDE)
- **Before**: `"배포 절차"` 질의 → 임베딩이 "배포"와 "절차"에만 집중 → "릴리즈 프로세스" 문서 유사도 낮음 → 검색 누락
- **After**: LLM이 가상 문서 생성: "배포 프로세스는 CI/CD 파이프라인을 통해 릴리즈됩니다…" → 가상 문서 임베딩에 "릴리즈", "CI/CD", "파이프라인" 포함 → 평균 벡터가 "릴리즈 프로세스" 문서와도 높은 유사도 → 검색 재현율 향상

### Phase 7.6: 문서 분류기 입력 범위 확대 (I-017)

**파일**: `src/context_loop/processor/classifier.py`

- **변경 전**: `content[:2000]`으로 앞 2000자만 LLM에 전달. 긴 문서의 후반부 구조(아키텍처 다이어그램, 엔티티 관계)를 볼 수 없어 잘못된 분류 발생.
- **변경 후**: 짧은 문서(4000자 이하)는 전문, 긴 문서는 시작/중간/끝 구간에서 균등 샘플링(각 1300자, 총 ~4000자).
- **주요 변경점**:
  - `_sample_content()`: 시작(1300자) + 중간(1300자) + 끝(1300자) 샘플링. `[Beginning]/[Middle]/[End]` 레이블 포함.
  - `_USER_PROMPT_TEMPLATE`: 고정 레이블 → 동적 `content_label`.
  - 4000자 이하 문서는 전문 전달 (하위 호환).
- **테스트**: 8개 신규 (샘플링 6 + 통합 2), 기존 포함 248개 통과
- **설계 결정**: D-023

## 주요 개선 효과 (Phase 7.6)

### 전체 문서 구조 기반 분류
- **Before**: 앞 2000자가 서술형 배경 → `chunk`로 분류. 하지만 후반에 아키텍처/엔티티 관계 포함 → hybrid가 맞음
- **After**: 시작(배경) + 중간(본문 핵심) + 끝(아키텍처) 모두 분석 → `hybrid`로 정확 분류
