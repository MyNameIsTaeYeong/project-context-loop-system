## 1. 변동성 측정 인프라 (`54f9d49`)

### 동기

단일 골드셋 1회 측정은 두 가지 노이즈를 분리하지 못함:
1. **샘플링 변동** — `stratified_sample` 이 무작위로 N개 청크를 고름
2. **LLM 확률성** — Generator/Judge 의 stochasticity

이 노이즈가 변경 효과보다 크면 "I-046 이 효과 있다" 같은 결론을 단일 측정으로
내릴 수 없음. 해결: 같은 source_type 으로 시드만 다른 N개 골드셋을 만들고
모두 채점해서 **mean ± std** 를 보면 노이즈 위에서 효과가 분리됨.

### Build 변경 (`scripts/build_synthetic_gold_set.py`)

- `--n-gold-sets N` (기본 1). N=1 이면 기존 동작 유지
- N>1 일 때 시드는 `seed, seed+1, …, seed+N-1` 로 자동 분배
- 출력 경로에 `_001`, `_002` … 접미사 자동 부여 (3자리 zero-pad, N>500 이면
  너비 확장). `--output eval/gold_sets/git_code.yaml --n-gold-sets 5` →
  `git_code_001.yaml` … `git_code_005.yaml`
- 각 빌드는 store/clients 를 재초기화하지 않고 동일 settings 로 순차 실행

### Eval 변경 (`scripts/eval_search.py`)

- `--gold-set` (단일) vs `--gold-set-glob "…*.yaml"` (다중) — `add_mutually_exclusive_group`
- 글롭 모드에서 stores/clients/judge 는 **한 번만 초기화**, N개 골드셋에 공유
- 잡별 출력: `{label}_{gold_set_stem}.{csv,summary.json}`
- 잡 1개 실패해도 다음 골드셋 계속 진행 — aggregate 에서 실패 잡 제외
- 신규 `{label}.aggregate.summary.json` — mean/std/min/max/n + per-gold-set
  세부 + gold_set 경로 추적

### Metrics 변경 (`src/context_loop/eval/metrics.py`)

`aggregate_with_variance(per_run_summaries)`:
- 여러 골드셋의 `aggregate` 출력을 받아 메트릭별 mean/std/min/max/n 산출
- 표본 표준편차 (ddof=1) 사용 — N=5 같은 작은 N 에서 편차 과소추정 방지
- n=1 이면 std=0, 결측 키는 가진 잡만 모아 통계 (judge 모드 잡 섞임 등)

### 판정 기준

`baseline.aggregate.summary.json` vs `after.aggregate.summary.json` 비교 시:
**`|mean Δ| > max(std_baseline, std_after)` 이면 통계적으로 유의미한 변화.**
단일 측정의 0.05 차이를 무비판적으로 "개선" 으로 보지 않음.

## 2. I-046 측정 워크플로 확정 — SQL hash 우회 방식

### 문제 — 인과 관계가 깨진 상태

I-046 의 변경(`ce75681`, `0a8a364`) 이 이미 머지됐는데 baseline (단일뷰) 측정이
없음. 단순 비교 시 다음 함정:

- 이미 멀티뷰 코드로 인덱싱됐다면 baseline 을 받을 수 없음
- 단순히 main 으로 checkout → 인덱싱은 hash short-circuit 으로 skip 될 수 있음
- 전체 데이터 디렉토리를 비우면 다른 source 데이터(confluence 등) 까지 손실

### 핵심 발견 4가지 (코드 추적으로 검증)

#### (a) `doc_id` 는 stable

- `documents` 테이블 `UNIQUE(source_type, source_id)`
- `source_id = file_info.relative_path` (`git_repository.py:358`, git_code 케이스)
- 재인덱싱은 `update_document_content()` 호출 → `UPDATE documents WHERE id=?`
  → **`documents.id` 절대 안 바뀜** (DELETE+INSERT 경로 없음)

#### (b) Hash short-circuit 함정

`git_repository.py:386-387`:
```python
if existing["content_hash"] == file_info.content_hash:
    return {**existing, "created": False, "changed": False}
```
`coordinator.py:171` 의 `if doc_result.get("changed"):` 분기 때문에 hash 같으면
**파이프라인 자체가 호출 안 됨** → vector_store 가 그대로 → 멀티뷰로 교체 안 됨.

#### (c) AUTOINCREMENT + 다른 source 존재 → 부분 삭제+재인덱싱 불가

`documents.id INTEGER PRIMARY KEY AUTOINCREMENT`. 다른 source 가 이미 들어있는
상태에서 git_code 만 DELETE 한 뒤 재인덱싱하면 `sqlite_sequence` 는 이전 max
값에서 시작 → 새 git_code 들은 **다른 doc_id** 로 발급됨. 골드셋의
`relevant_doc_ids` 무효화.

#### (d) `pathlib.Path.rglob` 비결정성 — 사용자 환경에선 무관

`git_repository.py:306` 의 `clone_dir.rglob("*")` 가 filesystem 의존 비결정적.
하지만 (c) 때문에 어차피 "전체 퍼지 + 재인덱싱" 시나리오 자체가 사용자 환경에서
불가능 — `sorted()` 패치는 이번 측정과 무관 (미래 다른 시나리오에서 가치는 있음).

### 최종 워크플로 — SQL hash 깨기

검토한 셋 (A: SQL 한 줄, B: `--force-reindex` 코드 옵션, C: UI 통합) 중
사용자 패턴 (일회성 평가, 웹 sync 버튼 사용) 에 맞춰 **A 선택**.

```bash
# 1) baseline 인덱싱 ID 스냅샷 (sanity check 용)
sqlite3 ~/.context-loop/data/metadata.db \
  "SELECT id, source_id FROM documents WHERE source_type='git_code' ORDER BY id" \
  > /tmp/ids_baseline.txt

# 2) 골드셋 빌드 + baseline 평가 (현재 단일뷰 인덱싱 상태에서)
python scripts/build_synthetic_gold_set.py \
  --source-types git_code --seed 42 --n-gold-sets 5 \
  --output eval/gold_sets/git_code.yaml
python scripts/eval_search.py \
  --gold-set-glob "eval/gold_sets/git_code_*.yaml" --label baseline

# 3) hash 깨기 — short-circuit 우회
sqlite3 ~/.context-loop/data/metadata.db \
  "UPDATE documents SET content_hash='_force_' WHERE source_type='git_code'"

# 4) 웹 git sync 버튼 클릭 (또는 CLI 인덱싱)
#    → hash 다름 → update_document_content() 호출 (id 보존)
#    → process_document() → chunks/vector/graph 전부 멀티뷰로 재구성

# 5) sanity check — doc_id 보존 + force 잔존 없음
sqlite3 ~/.context-loop/data/metadata.db \
  "SELECT id, source_id FROM documents WHERE source_type='git_code' ORDER BY id" \
  > /tmp/ids_after.txt
diff /tmp/ids_baseline.txt /tmp/ids_after.txt   # 차이 0줄이어야 함

sqlite3 ~/.context-loop/data/metadata.db \
  "SELECT COUNT(*) FROM documents
   WHERE source_type='git_code' AND content_hash='_force_'"   # 0 이어야 함

# 6) multiview 평가 — 같은 골드셋
python scripts/eval_search.py \
  --gold-set-glob "eval/gold_sets/git_code_*.yaml" --label multiview
```

### 검증 — 파이프라인 7단계 추적

hash 깨고 sync 트리거 시 실제로 무엇이 일어나는지 코드로 단계별 확인:

1. **Sync 시작 & 파일 수집** — `web/api/git_sync.py:277 _run_sync()` →
   `CoordinatorAgent.run_and_store()`, git pull + 파일 listing + hash 계산
2. **Hash 비교 → documents UPDATE** — `git_repository.py:386` mismatch 감지 →
   `update_document_content()` 호출 (`UPDATE … WHERE id=?`)
3. **파생 데이터 완전 삭제** — `reprocessor.py:84 start_reprocessing()` →
   `delete_derived_data(doc_id)` → chunks/graph_nodes/graph_edges DELETE
4. **AST 파싱 + 청킹** — `ast_code_extractor.py` 언어별 파싱 → 함수/클래스
   단위 청크 + meta_text (식별자 + signature + docstring) 생성
5. **멀티뷰 임베딩** — `pipeline.py:140-177`, body 뷰 + meta 뷰 두 임베딩 생성
6. **저장 (chunks + vector + graph)** — `vector_store.delete_by_document` →
   `add_chunks` (#body/#meta 두 row), `meta_store` 청크 INSERT,
   `graph_store.save_graph_data` (import 관계)
7. **완료 기록** — `complete_reprocessing()` → `processing_history` 'reprocessed'
   완료 + storage_method='hybrid'

→ `documents.id` 보존, 청크 내용/임베딩/그래프 전부 새 멀티뷰 코드로 완전
재구성. 측정 인과가 깨끗하게 분리됨.

### confluence_mcp 동등성

`mcp_confluence.py:1010` 의 hash 비교 패턴이 `git_repository.py:386` 과 동일.
같은 force-reindex 방식 적용 가능. 단:
- **MCP 서버 가용성 필수** — `/api/confluence-mcp/health` 먼저 확인
- 일부 페이지 실패 격리 가능 — sync 후 `content_hash='_force_'` 잔존 카운트
  확인 필요
- I-046 은 git_code 만의 변경이라 confluence_mcp 의 baseline=after 는 거의
  동일할 것 (멀티뷰가 이미 적용돼 있던 분기)

## 검토했으나 선택 안 한 옵션

| 옵션 | 채택 안 한 이유 |
|---|---|
| 각 인덱싱 상태별 별도 골드셋 | A/B 타당성 잃음 (다른 시험지). 두 측정의 차이가 시스템 개선인지 골드셋 난이도 차인지 구분 불가 |
| 전체 data 퍼지 + 재인덱싱 | 다른 source 데이터(confluence) 도 잃음. AUTOINCREMENT + rglob 비결정성으로 doc_id 보장 X |
| git revert + 임시 단일뷰 재인덱싱 | 재인덱싱 2회 비용, 작업 복잡 |
| `--force-reindex` CLI/웹 옵션 추가 | 1회성 측정엔 over-engineered. 향후 비슷한 측정이 정례화되면 고려 |
| 골드셋 라벨을 `source_id` 기반으로 schema 변경 | 가장 robust 하지만 코드 작업 크고 이번 측정 범위 초과 |
