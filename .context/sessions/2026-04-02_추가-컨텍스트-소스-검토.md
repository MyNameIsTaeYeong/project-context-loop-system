# 추가 컨텍스트 소스 검토 — Git 코드 기반 컨텍스트 구축 방안

- **일시**: 2026-04-02
- **목적**: Confluence 문서만으로는 부족한 사내 컨텍스트를 보완하기 위한 추가 소스 검토 및 구현 방향 결정

---

## 1. 현재 상태 분석

현재 시스템은 3가지 소스만 지원:
- **Confluence** (REST API / MCP Client): 구현 완료, 문서 중심이라 코드/설계 의도 부재
- **파일 업로드**: 구현 완료, 수동이라 최신성 보장 불가
- **직접 편집**: 구현 완료, 수동 작성 의존

**한계**: 실제 개발 코드, 커밋 히스토리, PR 리뷰 등 기술적 맥락이 시스템에 없음.

## 2. 추가 컨텍스트 소스 후보 평가

### 소스별 자동화·최신성 평가

| 소스 | 자동 수집 | 변경 감지 | 사람 개입 | 최신성 |
|------|----------|----------|----------|--------|
| Git 코드 | `git pull`만으로 전체 동기화 | commit hash 비교 | 거의 없음 | 커밋 즉시 반영 가능 |
| Confluence | REST API 가능 (구현됨) | content_hash 비교 | 거의 없음 | 동기화 주기 의존 |
| Jira | REST API 가능 | updated 필드 비교 | 거의 없음 | 실시간 가능 |
| Slack/Teams | API 가능, 노이즈 심함 | 메시지 timestamp | 필터링 규칙 필요 | 실시간 |
| DB 스키마 | `SHOW CREATE TABLE` 등 | DDL hash 비교 | 거의 없음 | 쿼리 즉시 |
| API 명세 (OpenAPI) | Git에 포함 | Git과 동일 | 없음 | Git과 동일 |
| CI/CD 로그 | API (GitHub Actions 등) | run_id 기반 | 없음 | 실행 즉시 |

### 우선순위 (ROI 기준)

| 순위 | 소스 | 이유 | 난이도 |
|------|------|------|--------|
| 1 | **Git Repository (코드 + 커밋 + PR)** | 하나의 접점에서 가장 많은 종류의 기술 컨텍스트 추출 가능. 사내 기술 컨텍스트의 60-70% 커버 | 중 |
| 2 | **Jira/이슈 트래커** | 요구사항-코드 연결, 진행 상황 추적 | 중 |
| 3 | **API 명세** | 자동 파싱 가능, Git에 포함되어 있으면 1순위와 동시 해결 | 하 |
| 4 | **DB 스키마** | 도메인 모델의 실체, 그래프 저장에 적합 | 하 |
| 5 | **Slack/Teams** | 가치 높지만 노이즈 필터링 복잡 | 상 |

## 3. 코드 → 컨텍스트 변환 방식 비교

### 접근 A: 코드 → LLM 문서 생성 → 기존 파이프라인으로 Chunking/Graph

```
VPC 코드 (.tf, .py) → LLM이 "VPC 아키텍처 문서" 작성 → 마크다운 Chunking → 벡터/그래프
```

### 접근 B: 코드 → AST 파싱 → 직접 Chunking/Graph

```
VPC 코드 (.tf, .py) → AST 파싱 → 함수/클래스 단위 Chunk → 벡터/그래프
```

### 상세 비교

| 기준 | A: 코드→문서→컨텍스트 | B: 코드→직접 컨텍스트 |
|------|----------------------|---------------------|
| **"왜" 정보** | LLM이 설계 의도를 추론하여 포함 | 코드만으로는 "왜"가 없음 |
| **자연어 검색** | 높은 매칭률 | 변수명/함수명에만 의존 |
| **정확성** | LLM 환각 위험 | 100% 정확 |
| **최신성 유지** | 코드 변경 시 문서 재생성 필요 (LLM 비용) | re-chunking만 하면 됨 (저비용) |
| **비개발자 접근성** | 높음 (자연어 문서) | 낮음 (코드 읽기 필요) |
| **관계 추출** | 의미적 관계 ("NAT GW는 프라이빗 서브넷용") | 참조 관계만 (`subnet_id = ...`) |
| **처리 비용** | 높음 (파일당 LLM 호출) | 낮음 (파싱만) |
| **기존 파이프라인 재사용** | 100% 재사용 가능 | 코드 전용 chunker 필요 |

### 결론: 하이브리드 (A + B) 채택 (D-025)

- **접근 A (code_doc)**: 자연어 검색, "왜" 컨텍스트, 비개발자 접근용. `source_type = "code_doc"`. 기존 chunker + graph_extractor 그대로 사용.
- **접근 B (git_code)**: 정확한 코드 확인, 환각 검증, 개발자 참조용. `source_type = "git_code"`. 파일 단위 저장.

## 4. code_doc ↔ git_code 연결 문제 (D-026)

**문제**: `code_doc` 1개가 여러 소스 파일을 참조할 수 있음. 기존 스키마로는 연결 불가.

### 검토한 방법 3가지

| 방법 | 정확성 | 구현 비용 | 다중 파일 지원 |
|------|--------|----------|--------------|
| 1. `source_id` 매칭 | O | 매우 낮음 | X — 1파일=1문서일 때만 |
| 2. `document_sources` 연결 테이블 | 매우 높음 | 중간 | O — N:M 지원 |
| 3. 벡터 재검색 | 낮음 | 낮음 | O — 하지만 부정확 |

### 결론: 방법 2 채택 — `document_sources` 테이블

```sql
CREATE TABLE document_sources (
    doc_id        INTEGER REFERENCES documents(id),
    source_doc_id INTEGER REFERENCES documents(id),
    file_path     TEXT,
    PRIMARY KEY (doc_id, source_doc_id)
);
```

**이유**: 실제 사내 코드에서 LLM 문서 1개가 단일 파일만 참조하는 경우는 드묾.
VPC 관련 코드만 해도 `vpc.tf`, `subnets.tf`, `route_tables.tf` 등 여러 파일에 걸쳐 있고,
LLM이 이것들을 종합하여 하나의 문서를 만드는 것이 자연스러움.

**검색 시 동작**:
1. `code_doc` chunk 반환 → `document_id` 확인
2. `document_sources`에서 `source_doc_id` 조회
3. 원본 코드 파일들 조회하여 LLM에 함께 제공 (문서=이해, 코드=검증)

## 5. 자동화 관점에서의 Git 소스 평가

**핵심 질문**: "Git 소스 기반이 자동화·최신성에서 최선인가?"

**답**: Git이 "최선의 단일 소스"인 것은 맞음. 하나의 접점에서 가장 많은 종류의 컨텍스트를 자동 추출 가능:
- 소스 코드 → "무엇이 있는가"
- 커밋 히스토리 → "왜 그렇게 했는가"
- PR/리뷰 → "설계 결정의 맥락"
- CODEOWNERS → "누가 무엇을 알고 있는가"
- IaC/OpenAPI/migration 파일 → 인프라, API, 스키마

**하지만 Git만으로 부족한 영역**:
- 진행 중인 업무 상황 → Jira 필요
- 비공식 의사결정 → Slack 필요
- 런타임 상태/장애 → 모니터링/로그 필요
- 비개발 문서 → Confluence/Notion 필요

**실용적 결론**: Git 먼저 구현 (가성비 최고) → Jira 두 번째 → 이 둘로 80-90% 커버 가능.
