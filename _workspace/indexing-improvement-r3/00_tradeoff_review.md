# R3 검토 — 임베딩 청크 단위를 문서 단위로 전환할 때의 트레이드오프

## 결론을 먼저

> **그대로 "청크 → 문서 1벡터"로 직역하면 5가지 사용자-체감 기능이 깨진다.** 옵션 B(섹션 단위 + 작은 문서는 1벡터)가 합리적이지만, 사내 임베딩 모델의 컨텍스트 한도를 확인해야 결정 가능.

## 임베딩 청크가 결합된 다운스트림 5곳

| # | 위치 | 청크가 제공하는 기능 | 문서 1벡터로 합치면 |
|---|------|-------------------|------------------|
| 1 | `_search_chunks` (`context_assembler.py:170`) — 벡터 유사도 검색 | 큰 문서 안에서 query와 닿는 **부분**을 정밀 매칭 | 큰 문서 전체가 1 벡터로 평균화 → 부분 매칭 정밀도 ↓ |
| 2 | `_format_chunk_results` (`context_assembler.py:236-239`) + `assemble_context_with_sources` (`context_assembler.py:413-416`) — **출처 라벨 "(섹션: …)"** | 청크 metadata 의 `section_path` 를 사용자에게 보여줌 | `section_path` 빈 문자열 → **"섹션: …" 출처 표시 사라짐** (사용자가 최근 PR #14 에서 출처 표시 강화 요청한 영역) |
| 3 | `mcp.context_max_tokens=4096` + `max_context_chunks=10` | 답변용 컨텍스트로 청크 10개를 4096토큰 안에 모음 | 문서 10개를 그대로 결합하면 평균 50K+ 토큰 → 가드 즉시 충돌 |
| 4 | `web/api/documents.py:135` + `tab_chunks.html` — 대시보드 **청크 탭** | 문서 상세 화면에서 청크 N개를 나열 (`body` / `body + meta` 멀티뷰 시각화) | "Chunk #0" 하나만 표시 — **탭 자체가 무의미해짐** |
| 5 | reranker (`reranker.py:46`) — query × document 쌍 점수 | 청크 N개를 각각 query 와 평가 → top-k 추림 | 문서 1개 vs query 1개 평가 = reranker 의미 약화. 큰 문서일수록 정밀도 ↓ |

위 5개 중 **#2(출처 라벨)와 #4(청크 탭)는 사용자-체감 회귀**이고, **#3(MCP 토큰 폭발)은 운영 차단**, **#1·#5는 검색 품질 저하**.

## "청킹" 의 진짜 강제 요인 (이번에야 분명)

R2 분석에서 확인했듯 청크 분할은 3종이 섞여 있고 이번 라운드는 **chunker (#1)** 만 해당:

| 분할 | 강제 요인 | 256K LLM 으로 사라지는가 |
|------|----------|--------------------|
| `chunker` (512 토큰) | (a) **임베딩 모델 토큰 한도** (사내 모델 미상), (b) **벡터 검색 정밀도** (큰 문서의 부분 매칭), (c) **출처 라벨**, (d) **MCP 응답 토큰 가드** | ❌ — LLM 컨텍스트와 무관, 임베딩/검색의 독립 제약 |
| `extraction_unit` | LLM 호출 단위 | ✅ R2 에서 처리됨 |
| `ast_code_extractor.to_chunks` | 코드 식별자 검색 정밀도 | ❌ 의미적 분할 |

→ **"청크사이즈로 나누는 것"의 사라지지 않는 이유 4가지** 가 남아있다. LLM 한도와는 완전히 다른 차원의 제약.

## 옵션 비교

### 옵션 A: 청크 = 문서 (사용자 요청의 직역)

| 측면 | 영향 |
|------|------|
| 인덱싱 단순화 | ★★★ |
| 임베딩 호출 비용 | -90% |
| 검색 정밀도 | ⚠️ 큰 문서 정밀도 -30~50% (정량 미측정) |
| 출처 라벨 | ❌ "(섹션: …)" 사라짐 |
| 대시보드 청크 탭 | ❌ 무의미 |
| MCP 응답 가드 | ❌ context_max_tokens 동반 조정 필수 |
| reranker | ⚠️ 정밀도 ↓ |
| 임베딩 모델 한도 | ❓ 사내 모델 한도 확인 필수 |

### 옵션 B: **섹션 단위 청킹** (1500~3000 토큰 목표, 코드는 파일/심볼 유지)

각 `Section` 을 하나의 청크로 (`extraction_unit` 의 응축 로직과 유사한 입자도). R2 의 ExtractionUnit 입자도(1500 토큰)에 정렬.

| 측면 | 영향 |
|------|------|
| 인덱싱 단순화 | ★★ (분할 정책 1개 줄어듦) |
| 임베딩 호출 비용 | -60~70% |
| 검색 정밀도 | ✅ 거의 동일 (큰 문서도 섹션 단위는 의미 보존) |
| 출처 라벨 | ✅ `section_path` 유지 — UX 무변경 |
| 대시보드 청크 탭 | ✅ 섹션 단위로 깔끔하게 표시 (오히려 가독성 향상) |
| MCP 응답 가드 | △ `context_max_tokens` 를 8K~16K 로 약간 상향 |
| reranker | ✅ 섹션 = 의미 단위, 작동 적정 |
| 임베딩 모델 한도 | ✅ 사내 모델이 8K 만 지원해도 대부분 섹션 < 3K 안전 |

### 옵션 C: **하이브리드** — 작은 문서(<X 토큰) = 1 벡터, 큰 문서 = 섹션 단위

| 측면 | 영향 |
|------|------|
| 인덱싱 단순화 | △ (분기 추가) |
| 임베딩 호출 비용 | -80% |
| 검색 정밀도 | ✅ 큰 문서는 섹션 유지, 작은 문서는 1벡터로 충분 |
| 출처 라벨 | ⚠️ 작은 문서는 section_path 빈 문자열 — 부분 손실 |
| 대시보드 청크 탭 | △ 작은 문서는 청크 1개 — 약간 무의미 |
| MCP 응답 가드 | △ |
| 임베딩 모델 한도 | ✅ 가드로 자동 폴백 |

## 권고

**옵션 B (섹션 단위 청킹)** — 사용자 의도("청크사이즈로 나누는 것을 없애고")와 다운스트림 보존을 모두 만족하는 최적해. "임의 토큰 분할 (512)" 은 사라지고, 분할 단위가 의미 단위(섹션) 로 통일된다.

핵심 통찰:
- 사용자의 진짜 불편은 "**의미와 무관하게 512토큰에서 잘리는 것**" 일 가능성이 큼 (R1 F-04 "overlap 토큰→블록 단위", F-12 "nested 헤딩 본문 누락" 등이 이미 그 시그널).
- 섹션 단위 = ExtractionUnit 입자도와 정렬 → R2 의 LLM 호출 단위와 임베딩 단위가 같은 입자도로 통일되어 인덱싱 멘탈 모델 단순화.

## 결정 전에 사용자 확인이 필요한 항목

1. **사내 임베딩 모델 컨텍스트 윈도우** — 큰 섹션/문서가 한 번에 임베딩 가능한지 (8K? 32K? 256K?)
2. **검색 유즈케이스 우선순위** — "어떤 문서가 관련 있나" (문서 단위 충분) vs "큰 문서의 어느 부분에 답이 있나" (섹션/청크 필요)
3. **출처 라벨 "(섹션: …)" 유지 의향** — 최근 PR #14 에서 강화한 부분
