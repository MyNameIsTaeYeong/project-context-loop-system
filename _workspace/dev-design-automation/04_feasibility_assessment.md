# 04. 시나리오별 타당성 평가

> 평가 축: **구현 가능성**(building block 준비도) · **필요 작업량**(S=수일 / M=1~2주 / L=수주+) · **기대 효과**(★) · **그래프품질 의존도** · **주요 리스크**.
> 근거: 01 인벤토리 + 백그라운드 3개 분석가 교차검증. git_code AST 그래프(D-036)는 결정론·고신뢰.

## 4.1 공통 선행 enabler (시나리오 전반의 비용/효과를 바꾸는 항목)

| Enabler | 내용 | 작업량 | 영향 |
|---------|------|--------|------|
| **E1. CLI 복원** | `cli.py` 신규(`context-loop index/serve/review/...` 서브커맨드). 현재 엔트리포인트 깨짐 | S | 모든 자동화의 배치/CI/cron 트리거 표준화. **모든 시나리오 운영화의 전제** |
| **E2. 코퍼스 인덱싱 채우기** | 대상 레포(git_code) + 사내 문서(confluence) 실제 인덱싱 | S~M | 데이터가 없으면 어떤 시나리오도 무의미. **데이터 선행** |
| **E3. 호출(call) 그래프 추가** | `ast_code_extractor`에 `ast.Call` 분석 → `calls` relation | M | D-2/D-7/S-2/S-10 정밀도 동시 향상. **단일 고레버리지** |
| **E4. 그래프 스냅샷/diff 인프라** | 인덱싱 시점 그래프 버전 보존·비교 | L | S-4(문서 자동 업데이트), 변경기반 시나리오의 전제 |

> **권고:** E1·E2는 거의 모든 시나리오의 전제이므로 0순위. E3은 코드 구조 자동화 전반의 정밀도를 끌어올리는 레버리지. E4는 고비용이라 변경기반 자동화(S-4) 진입 전까지 보류.

## 4.4 타당성 핵심 결론

1. **즉시 착수 가능(작업량 S, 의존 낮음):** D-1, D-8, D-9, S-1, S-9, S-7. — 대부분 기존 역량 조합 + 경량 래퍼. E1·E2만 갖추면 수일 내 PoC.
2. **고효과·중간 작업량:** D-2(코드 리뷰), S-2(영향도), D-4·S-10(부채/안티패턴). — E3(call 그래프)이 정밀도를 크게 끌어올림.
3. **고비용·보류 권고:** S-4(문서 자동 업데이트, E4 필요), S-5(drift 검증, 그래프 품질 한계로 자동화 부적합). 현 단계에선 **검수 보조** 위치로만.
4. **데이터·인프라가 코드보다 큰 제약:** building block은 충분히 성숙. 정작 막는 것은 (a) 코퍼스가 실제로 채워졌는지(E2), (b) 자동화 트리거 표준(E1), (c) 코드 정밀도 한 단계(E3)다.
