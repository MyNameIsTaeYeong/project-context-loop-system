🔄 Context Loop

사내 지식을 인덱싱해 MCP로 AI 툴(Claude 등)에 컨텍스트로 제공하는 시스템. 웹 대시보드는 그 지식이 백엔드에 어떻게 저장됐는지 들여다보는 도구입니다.

이 프로젝트는 무엇을 위한 건가요?

회사의 지식은 Confluence, 코드 저장소, 개인 메모 등 여러 곳에 흩어져 있습니다. Context Loop는 이 분산된 지식을 한곳에 인덱싱해, MCP 프로토콜로 Claude 같은 AI 툴에 구조화된 컨텍스트로 제공하는 것을 주 목적으로 합니다.

웹 대시보드는 그 지식이 백엔드(벡터·그래프·메타데이터 저장소)에 실제로 어떻게 저장됐는지를 눈으로 확인하고 점검하는 도구입니다. 즉, 인덱싱 결과를 들여다보고 다듬는 관찰창이며, 실제 활용은 AI 툴에서 MCP를 통해 이뤄집니다.

🚀 어떻게 사용하나요?

주 사용처는 AI 툴에서 MCP로 사내 지식을 불러오는 것입니다. 흐름은 다음과 같습니다.

1
지식 인덱싱

Confluence MCP·Git 코드·파일 업로드·직접 작성으로 문서를 등록하면 자동으로 청킹·임베딩·그래프화됩니다.

2
대시보드로 확인

무엇이 어떻게 저장됐는지(청크·그래프·메타데이터)를 웹에서 점검하고 필요하면 다듬습니다.

3
AI 툴에 MCP 연결

Claude Code·opencode 등의 설정에 Context Loop MCP 서버를 등록합니다(로컬 stdio 또는 원격 SSE).

4
AI에서 질의

AI가 MCP 도구로 사내 지식을 자동으로 컨텍스트에 포함해 답변합니다.

🔗 AI 툴에 연결하기

사내 PC에 떠 있는 MCP 서버에 AI 툴에서 그 주소로 원격 연결합니다.

opencode 설정 (opencode.json):

{
  "mcp": {
    "context-loop": {
      "type": "remote",
      "url": "http://70.50.170.15:3001/sse",
      "enabled": true
    }
  }
}

연결되면 AI가 다음 4가지 도구를 사용합니다.

  • search_context — 질의로 관련 지식(벡터+그래프)을 검색·조립
  • list_documents — 인덱싱된 문서 목록 조회(소스/상태 필터)
  • get_document — 특정 문서를 원본/청크/그래프 형식으로 조회
  • get_graph_context — 특정 엔티티 중심의 그래프 관계 탐색

🔧 Git 코드는 어떻게 임포트되나요?

설정한 Git 저장소의 코드를 심볼 단위로 인덱싱합니다. 저장소는 URL·브랜치·상품(파일 경로 스코프) 단위로 여러 개 설정할 수 있습니다.

1
저장소 동기화

저장소를 clone/pull 하고 커밋 해시를 기록(증분 대비)

2
파일 수집

상품 경로(glob)·제외 패턴 + 지원 확장자·크기 한도로 필터링 (node_modules·build 자동 제외)

3
심볼 추출 (AST)

함수·클래스·메서드를 정적 분석으로 추출. Python=ast, Go·Java·TS·JS 지원. LLM 호출 없음(비용 0)

4
멀티뷰 임베딩

심볼 1개 = 청크 1개. 본문(코드)과 메타(이름·시그니처)를 별도 view로 임베딩

5
import 그래프

파일↔모듈 import, 클래스→메서드 contains 관계 추출. 같은 모듈 import는 한 노드로 병합

6
증분 저장

content_hash 비교로 변경된 파일만 재처리(Delete & Recreate)

🛠️ 인덱싱은 어떻게 이루어지나요?

문서가 들어오면 다음 6단계를 거쳐 검색 가능한 인덱스로 변환합니다.

1
수집

원본과 content_hash를 함께 보관

2
전처리

HTML→마크다운, 구조 정규화 (코드는 AST 분석)

3
청킹

토큰 단위(문서) 또는 심볼 단위(코드)로 분할

4
임베딩

본문·메타·가상질문을 각각 별도 view로 임베딩

5
그래프 추출

엔티티·관계 추출, 표기 변형은 병합

6
저장

ChromaDB·NetworkX+SQLite·SQLite에 저장

원본이 바뀌면 content_hash 비교로 변경을 감지해, 기존 파생 데이터를 지우고 다시 만드는 Delete & Recreate 방식으로 재처리합니다.

💡 가상 질문 생성 (멀티 벡터 인덱싱)

임베딩 단계에서 LLM이 각 섹션마다 "이 섹션이 답할 수 있는 자연스러운 질문"을 미리 만들어 별도 view로 등록합니다(문서당 1회 호출, 섹션당 약 5개). 사용자의 자연어 질의가 본문보다 질문 형태의 임베딩과 더 가깝게 매칭되어 검색 재현율·정밀도가 향상됩니다. 생성된 질문은 문서 상세의 청크 탭에서 볼 수 있습니다.

🔍 검색은 어떻게 이루어지나요?

벡터 검색과 지식 그래프 탐색을 함께 수행해 하나의 컨텍스트로 조립합니다.

1
질의 임베딩

질의를 벡터로 변환 (선택: HyDE로 가상 답변 보강)

2
벡터 검색

ChromaDB 코사인 유사도 → 문서 단위 dedup → top-k

3
그래프 탐색

LLM 플래너가 엔티티 식별 → 깊이 1 양방향 BFS

4
리랭킹(선택)

cross-encoder로 재점수화해 정밀도 향상

5
병합·정렬

벡터+그래프 결과 통합, 중복 제거 후 유사도 정렬

검색 결과로 무엇을 돌려주나요?

MCP search_context는 포맷된 컨텍스트 텍스트(문자열)를 반환합니다. ## 관련 문서, ## 관련 그래프 컨텍스트, ## 그래프 연결 문서 등 섹션으로 구성되고, 항목마다 제목·유사도·섹션 경로(가상 질문 매칭 시 해당 질문)가 텍스트로 붙습니다. document_id 같은 구조화된 필드는 없으며, AI는 이 텍스트에서 근거를 읽습니다.
