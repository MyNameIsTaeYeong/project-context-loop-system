Context Loop

사내 지식을 LLM 컨텍스트로 변환·저장하고, 웹 대시보드와 MCP 서버로 관리하는 통합 지식 시스템입니다.

이 프로젝트는 무엇을 위한 건가요?

회사의 지식은 Confluence, 위키, 코드 저장소, 개인 메모 등 여러 곳에 흩어져 있습니다. Context Loop는 이렇게 분산된 지식을 한곳으로 자동 수집·처리하여, 웹 대시보드에서 시각적으로 관리하고 MCP 프로토콜을 통해 Claude 같은 사내 LLM 애플리케이션에 구조화된 컨텍스트로 제공합니다.

주요 기능

다양한 입력 경로 (4가지)

  • Confluence MCP 클라이언트 — MCP로 Confluence 문서를 검색·임포트
  • 파일 업로드 — 마크다운·텍스트·HTML 파일
  • 마크다운 직접 작성 — 내장 에디터(편집/미리보기 분할 뷰)
  • Git 코드 수집 — 저장소를 clone/pull 하여 코드 심볼 단위로 수집

웹 대시보드

  • 등록된 문서 관리 및 소스/상태별 필터링, 통계 확인
  • 지식 그래프 시각화 — 엔티티·관계 탐색, 병합된 노드 확인
  • 문서 상세 보기 — 원본 / 청크 / 그래프 / 메타데이터(처리 이력) 탭

연동과 동기화

  • 자동 동기화 — content hash로 원본 변경을 감지해 재처리(Delete & Recreate)
  • MCP 서버 — 사내 LLM 앱에 search_context, list_documents, get_document, get_graph_context 도구 제공

검색은 어떻게 이루어지나요?

질의가 들어오면 벡터 검색과 지식 그래프 탐색을 함께 수행해 하나의 컨텍스트로 조립합니다. 단계는 다음과 같습니다.

  1. 질의 임베딩 — 사용자 질의를 임베딩 벡터로 변환합니다. 선택적으로 HyDE를 켜면 질의로부터 가상의 답변 문서를 만들어 그 임베딩을 평균내어, 짧은 질의의 의미를 보강합니다.
  2. 벡터 검색 — ChromaDB에서 코사인 유사도로 가장 가까운 청크를 찾습니다. 본문·메타데이터·가상 질문이 각각 별도 view로 색인되어 있어, 필요 수의 몇 배를 넉넉히 가져온 뒤(over-fetch) 문서 단위로 중복을 제거하고 유사도 임계값 미만은 걸러 상위 top-k만 남깁니다. 가상 질문 view로 적중하면 "매칭 질문" 라벨이 함께 붙습니다.
  3. 그래프 탐색 — LLM 플래너가 질의 의도에 맞는 대상 엔티티·관계를 식별하고(plan_graph_search), 이름 일치(완전·부분) → 엔티티 임베딩 유사도 순으로 시작 노드를 찾습니다. 찾은 노드에서 깊이 1의 양방향 BFS로 직접 연결된 엔티티·관계를 모읍니다.
  4. 리랭킹(선택) — 리랭커가 켜져 있으면 전용 cross-encoder 모델로 벡터 결과를 다시 점수화하고, 점수 임계값을 넘는 상위 항목만 남겨 정밀도를 높입니다.
  5. 병합·정렬·보강 — 벡터 결과와, 그래프가 도달했지만 벡터가 놓친 문서의 본문을 합쳐 문서 단위로 중복 제거하고 유사도 순으로 정렬합니다. 필요 시 적중 청크를 그 문서의 전문으로 치환하는 Parent-document 검색, 코드 문서에 원본 소스를 덧붙이는 보강도 선택적으로 동작합니다.

검색 결과로 무엇을 돌려주나요?

검색은 ① 포맷된 컨텍스트 텍스트와 ② 구조화된 출처 목록을 함께 반환합니다.

  • 컨텍스트 텍스트 — ## 관련 문서, ## 관련 그래프 컨텍스트, ## 그래프 연결 문서, ## 원본 소스 코드(검증용) 등 여러 섹션으로 구성된 하나의 문자열로, 각 항목에 문서 제목·유사도·섹션 경로(가상 질문 매칭 시 해당 질문)가 붙습니다. 이 텍스트가 LLM에 그대로 전달됩니다.
  • 출처 목록 — 각 출처는 document_id, title, similarity(코사인 유사도), full_document(전문 치환 여부) 필드를 가지며, 그래프 탐색으로 찾은 엔티티·관계 참조도 함께 반환됩니다. 웹 질의응답에서는 답변 스트림과 별도로 이 출처가 전달되어 근거 문서를 표시할 수 있습니다.

이 조립 로직은 MCP search_context 도구와 웹 질의응답이 동일하게 사용합니다. (MCP 도구는 컨텍스트 텍스트를, 웹은 텍스트와 출처를 함께 돌려줍니다.)

어떤 원리로 구현되어 있나요?

6단계 인덱싱 파이프라인

문서가 들어오면 다음 단계를 거쳐 검색 가능한 인덱스로 변환합니다.

  1. 수집 — Confluence(MCP)·파일 업로드·마크다운 작성·Git 저장소 등에서 원본을 가져오고, 원문과 content_hash를 함께 보관합니다.
  2. 전처리 — HTML을 마크다운으로 변환하고 제목·섹션 같은 구조를 추출해 정규화합니다. 코드 소스는 이 단계에서 AST로 심볼(함수·클래스) 경계를 파악합니다.
  3. 청킹 — 정규화된 텍스트를 토큰 기준으로 분할하거나(문서), 함수·클래스 같은 심볼 단위로 분할합니다(코드). 분할 단위마다 안정적인 논리 청크 ID를 부여합니다.
  4. 임베딩 — 각 청크 본문을 임베딩하고, 아래의 가상 질문도 함께 임베딩하여 본문·메타데이터·질문을 각각 별도 view로 등록합니다(멀티 벡터 인덱싱).
  5. 그래프 추출 — 엔티티(노드)와 관계(엣지)를 뽑아냅니다. 코드는 AST 기반으로 import·호출 관계를, Confluence는 제목·섹션 계층의 링크 관계를, 일반 문서는 필요 시 LLM으로 의미 관계를 추출합니다. 같은 엔티티의 표기 변형은 정규화 키로 병합됩니다.
  6. 저장 — 청크·임베딩은 ChromaDB에, 엔티티·관계는 NetworkX + SQLite에, 문서·청크·처리 이력은 SQLite에 저장하여 벡터 검색과 그래프 탐색 모두에 대비합니다.

원본이 바뀌면 content_hash 비교로 변경을 감지해, 기존 파생 데이터(청크·그래프)를 지우고 다시 만드는 Delete & Recreate 방식으로 재처리합니다.

가상 질문 생성 (멀티 벡터 인덱싱)

임베딩 단계에서, LLM이 각 섹션마다 "이 섹션이 답할 수 있는 자연스러운 질문"을 미리 만들어 둡니다 (문서당 한 번 호출, 섹션당 약 5개 목표, 중복은 제거). 이 가상 질문들을 별도의 임베딩 view로 함께 등록하면, 사용자가 자연어로 던지는 질의가 본문보다 질문 형태의 임베딩과 같은 의미 공간에서 더 가깝게 매칭되어 검색 재현율과 정밀도가 향상됩니다. 생성된 질문은 문서 상세의 청크 탭에서 확인할 수 있습니다.

소스 타입별 최적화

  • Git 코드 — AST 정적 분석으로 함수·클래스를 추출(LLM 호출 없음, 비용 0)
  • Confluence 문서 — 제목·섹션 계층을 구조 기반 링크 그래프로 변환
  • 일반 문서 — 토큰 기반 청킹(필요 시 LLM 본문 추출로 의미 관계 보강)
