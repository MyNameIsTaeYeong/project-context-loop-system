Context Loop

사내 지식을 LLM 컨텍스트로 변환·저장하고, 웹 대시보드와 MCP 서버로 관리하는 통합 지식 시스템입니다.

이 프로젝트는 무엇을 위한 건가요?

회사의 지식은 Confluence, 위키, 코드 저장소, 개인 메모 등 여러 곳에 흩어져 있습니다. Context Loop는 이렇게 분산된 지식을 한곳으로 자동 수집·처리하여, 웹 대시보드에서 시각적으로 관리하고 MCP 프로토콜을 통해 Claude 같은 사내 LLM 애플리케이션에 구조화된 컨텍스트로 제공합니다.

원본 문서가 바뀌면 변경을 감지해 저장된 지식을 자동으로 갱신합니다 — 지식이 수집·가공·제공되고 다시 갱신되는 이 순환(loop)이 프로젝트 이름의 배경입니다.

주요 기능

다양한 입력 경로 (4가지)

  • Confluence MCP 클라이언트 — MCP로 Confluence 문서를 검색·임포트
  • 파일 업로드 — 마크다운·텍스트·HTML 파일
  • 마크다운 직접 작성 — 내장 에디터(편집/미리보기 분할 뷰)
  • Git 코드 수집 — 저장소를 clone/pull 하여 코드 심볼 단위로 수집

웹 대시보드

  • 등록된 문서 관리 및 소스/상태별 필터링, 통계 확인
  • 지식 그래프 시각화 — 엔티티·관계 탐색, 병합된 노드 확인
  • 문서 상세 보기 — 원본 / 청크 / 그래프 / 메타데이터(처리 이력) 탭

연동과 동기화

  • 자동 동기화 — content hash로 원본 변경을 감지해 재처리(Delete & Recreate)
  • MCP 서버 — 사내 LLM 앱에 search_context, list_documents, get_document, get_graph_context 도구 제공

검색은 어떻게 이루어지나요?

질의가 들어오면 벡터 검색과 지식 그래프 탐색을 함께 수행해 관련 컨텍스트를 조립합니다.

  1. 질의 임베딩 — 사용자 질의를 임베딩 벡터로 변환합니다. (선택적으로 HyDE로 가상 답변을 함께 임베딩해 검색을 보강합니다.)
  2. 벡터 검색 — ChromaDB에서 코사인 유사도로 가장 가까운 청크를 top-k로 찾고, 같은 문서가 여러 번 매칭되면 중복을 제거합니다.
  3. 그래프 탐색 — LLM 플래너가 질의 의도에 맞는 엔티티·관계를 식별하고, 이름 매칭과 엔티티 임베딩으로 시작 노드를 찾은 뒤 깊이 1의 BFS로 주변 노드를 탐색합니다.
  4. 리랭킹(선택) — 전용 리랭커(cross-encoder) 모델로 벡터 결과를 다시 점수화해 정밀도를 높입니다.
  5. 병합·정렬 — 벡터 결과와, 그래프가 도달했지만 벡터가 놓친 문서의 본문을 합치고, 중복 제거 후 유사도 순으로 정렬합니다. (필요 시 적중 청크를 문서 전문으로 치환하는 Parent-document 검색도 지원합니다.)

이렇게 조립된 컨텍스트는 출처 정보와 함께 반환되어, MCP search_context 도구와 질의응답 양쪽에서 동일하게 사용됩니다.

어떤 원리로 구현되어 있나요?

6단계 인덱싱 파이프라인

수집 → 전처리 → 청킹 → 임베딩 → 그래프 추출 → 저장의 단계를 거쳐, 원본을 검색 가능한 인덱스로 변환합니다.

가상 질문 생성 (멀티 벡터 인덱싱)

색인 단계에서 LLM이 각 섹션마다 "이 섹션이 답할 수 있는 자연스러운 질문"을 미리 만들어 둡니다 (문서당 한 번 호출, 섹션당 약 5개 목표). 이 가상 질문들을 별도의 임베딩 view로 함께 등록하면, 사용자가 자연어로 던지는 질의가 본문보다 질문 형태의 임베딩과 같은 의미 공간에서 더 가깝게 매칭되어 검색 재현율과 정밀도가 향상됩니다. 생성된 질문은 문서 상세의 청크 탭에서 확인할 수 있습니다.

소스 타입별 최적화

  • Git 코드 — AST 정적 분석으로 함수·클래스를 추출(LLM 호출 없음, 비용 0)
  • Confluence 문서 — 제목·섹션 계층을 구조 기반 링크 그래프로 변환
  • 일반 문서 — 토큰 기반 청킹(필요 시 LLM 본문 추출로 의미 관계 보강)
