# 과제계획서: Context Loop System

## 1. 과제 대상

**사내 지식 통합 관리 및 LLM 컨텍스트 제공 시스템 (Context Loop System)**

사내에 분산된 문서와 지식(Confluence, 파일, 수기 문서 등)을 자동으로 수집·분류·가공하여, 웹 대시보드를 통한 시각적 탐색과 MCP(Model Context Protocol) 서버를 통한 내부 LLM 애플리케이션 연동을 동시에 지원하는 통합 지식 관리 시스템을 개발한다.

---

## 2. 선정 배경

### 현황

- 사내 지식이 Confluence, 공유 드라이브, 개인 문서 등에 분산 저장되어 있어 필요한 정보를 찾는 데 과도한 시간이 소요됨
- LLM 기반 업무 도구(Claude, ChatGPT 등)의 도입이 확대되고 있으나, 사내 지식과의 연동이 부재하여 일반적인 답변만 생성 가능
- 문서의 생성·수정 이력이 체계적으로 관리되지 않아, 최신 정보와 과거 정보의 구분이 어려움

### 필요성

- 사내 LLM 애플리케이션이 조직의 맥락(Context)을 이해하고 정확한 답변을 생성하려면, 사내 지식이 구조화되어 접근 가능해야 함
- MCP 프로토콜 표준화에 따라, 다양한 LLM 도구에 사내 지식을 동일한 인터페이스로 제공할 수 있는 기반 인프라가 필요
- 문서의 자동 분류 및 최적 저장 방식 결정을 통해 지식 관리의 효율을 높일 필요가 있음

## 3. Pain Point

| 구분 | Pain Point | 영향 |
|------|-----------|------|
| **지식 분산** | 사내 지식이 Confluence, 파일 서버, 개인 노트 등에 흩어져 있어 통합 검색 불가 | 필요한 정보를 찾기 위해 여러 시스템을 개별 탐색해야 하며, 평균 정보 탐색 시간 과다 소요 |
| **LLM 맥락 부재** | 내부 LLM 도구가 사내 지식에 접근할 수 없어, 조직 맥락 없는 범용 답변만 생성 | LLM 도구의 실질적 업무 활용도 저하, 사용자 신뢰도 감소 |
| **수동 문서 가공** | 문서를 LLM에 활용하기 위해 수동으로 청킹, 임베딩 등의 전처리가 필요 | 지식 엔지니어링에 과도한 인력·시간 투입, 비효율적 운영 |
| **변경 감지 부재** | 원본 문서가 수정되어도 기존 가공 데이터와의 동기화가 이루어지지 않음 | 오래된 정보 기반의 잘못된 의사결정 유발 가능성 |
| **문서 최신성 불투명** | 특정 Confluence 문서가 최근 갱신됐는지, 누가 소유자인지 질의 결과에서 드러나지 않음 | LLM이 오래된 정보를 사실처럼 제시, 신뢰 저하 |
| **관계 정보 손실** | 단순 텍스트 검색으로는 문서 간, 개념 간의 관계를 파악할 수 없음 | 지식의 연결 구조를 활용한 심층 탐색 불가 |

## 5. 개선 방법

### 5.1 지식 수집 자동화

- **Confluence API 연동**: REST API를 통한 Space/Page 자동 수집 및 증분 동기화(content_hash 기반 변경 감지)
- **파일 업로드**: Markdown, 텍스트, HTML 파일의 드래그 앤 드롭 업로드 지원
- **직접 편집**: 웹 기반 마크다운 에디터를 통한 실시간 문서 작성·수정

### 5.2 지능형 문서 처리

- **LLM 기반 자동 분류**: 문서 내용을 분석하여 최적 저장 방식(Chunk/Graph/Hybrid)을 자동 결정
- **토큰 기반 청킹**: 설정 가능한 chunk_size와 overlap으로 텍스트를 의미 단위로 분할
- **엔터티·관계 추출**: LLM을 활용한 문서 내 엔터티 및 관계 자동 추출, 지식 그래프 구축

### 5.3 하이브리드 저장 구조

- **벡터 DB (ChromaDB)**: 시맨틱 검색을 위한 임베딩 기반 유사도 검색
- **그래프 DB (NetworkX + SQLite)**: 엔터티 간 관계 저장 및 탐색
- **메타데이터 DB (SQLite)**: 문서 메타데이터, 처리 이력, 청크 참조 관리

### 5.4 이중 인터페이스 제공

- **웹 대시보드**: 문서 관리, 그래프 시각화, RAG 기반 채팅 인터페이스
- **MCP 서버**: 내부 LLM 애플리케이션이 사내 지식을 조회할 수 있는 표준 프로토콜 인터페이스

### Phase 2: 부서 SSOT 전환 — "검색/조립 SSOT" 방향 (진행 중)

이 시스템을 부서의 Single Source of Truth로 **질의 진입점 관점**에서 먼저
확립한다. 쓰기는 Confluence에 유지하고, 본 시스템은 주기적 mirror + 검색·
조립·관측 계층으로 기능한다. 쓰기까지 이관하는 "완전 SSOT" 전환은 Phase 3
이후에 관측 데이터를 근거로 진행한다.

- [x] 특정 Confluence space 지속 import (증분 sync, `content_hash`·`version`)
- [x] 검색/인용 관측: `search_logs`·`search_citations` 테이블에 MCP·웹 질의와
      인용된 문서 ID를 기록 — hot/cold 문서 분류와 품질 리뷰 우선순위의 근거
- [x] 문서 staleness 신호: `source_updated_at` 보존 + fresh/aging/stale 버킷
      분류 (기본 90일/180일 임계값) — 오래된 정보 식별
- [x] Owner 메타데이터: Confluence `ownerId`를 별도 컬럼으로 보존 — stale
      알림·리뷰 요청의 수신자 경로
- [x] Sync 상태 대시보드: `sync_runs` 영속화 + `/api/sync/status` + 대시보드
      파셜 — "이 시스템이 Confluence만큼 최신인가" 사용자 확인 가능
- [ ] 엔터티 병합 테이블을 통한 중복 엔터티 통합
- [ ] LLM 분류기 프롬프트 최적화 (정확도·비용 개선)
- [ ] HTML→Markdown 변환 품질 향상 (markdownify 라이브러리 적용)
- [ ] 채팅 대화 이력 영속화
- [ ] 배포 패키징 및 설치 마법사

## 7. AI 기술 활용 계획

### 7.1 LLM (대규모 언어 모델) 활용

| 활용 영역 | 기술 | 설명 |
|----------|------|------|
| **문서 자동 분류** | LLM Classifier | 문서 내용을 분석하여 최적 저장 방식(chunk/graph/hybrid)을 자동 결정 |
| **엔터티·관계 추출** | LLM Graph Extractor | 문서에서 핵심 엔터티와 그들 간의 관계를 구조화하여 추출 |
| **지능형 그래프 탐색** | LLM Graph Search Planner | 사용자 질의 의도를 분석하여 최적의 그래프 탐색 경로를 계획 |
| **RAG 기반 Q&A** | Retrieval-Augmented Generation | 벡터 검색 + 그래프 탐색 결과를 결합하여 사내 지식 기반 답변 생성 |
| **컨텍스트 조합** | Context Assembler | 벡터 검색과 그래프 탐색 결과를 병합·중복 제거·관련도 순 정렬 |

### 7.2 임베딩 기술 활용

- **멀티 프로바이더 지원**: OpenAI, Anthropic, 로컬 모델, 커스텀 엔드포인트 등 다양한 임베딩 제공자 선택 가능
- **시맨틱 검색**: 텍스트 청크 및 그래프 엔터티에 대한 임베딩 생성으로 의미 기반 유사도 검색 구현
- **OpenAI 호환 엔드포인트**: vLLM, Ollama 등 자체 호스팅 모델과도 호환되는 인터페이스 설계

### 7.3 MCP (Model Context Protocol) 활용

- **표준 프로토콜 기반 LLM 연동**: Claude, ChatGPT 등 다양한 LLM 도구에서 동일한 인터페이스로 사내 지식 접근
- **4종 MCP Tool 제공**:
  - `search_context`: 질의 기반 컨텍스트 검색 및 조합
  - `list_documents`: 필터링 기반 문서 목록 조회
  - `get_document`: 원본/청크/그래프 형태의 문서 조회
  - `get_graph_context`: 엔터티 중심 그래프 탐색
- **stdio/SSE 전송 지원**: 로컬 및 원격 환경 모두 지원

## 8. 최적화 계획

### 8.1 성능 최적화

| 항목 | 현재 방식 | 최적화 방안 | 기대 효과 |
|------|----------|------------|----------|
| **문서 동기화** | 전체 삭제 후 재생성 (Delete & Recreate) | 증분 동기화 + content_hash 기반 변경 감지 적용 완료, 향후 diff 기반 부분 업데이트 검토 | 동기화 시간 50% 이상 단축 |
| **임베딩 생성** | 단건 처리 | 배치 처리 및 캐싱 도입 | API 호출 비용 절감, 처리 속도 향상 |
| **그래프 검색** | 전체 그래프 로드 | LLM 기반 탐색 계획 수립 → 필요 영역만 탐색 (적용 완료) | 검색 정확도 향상, 불필요한 연산 감소 |
| **LLM 호출 비용** | 분류·추출 시 매회 LLM 호출 | 프롬프트 최적화, 결과 캐싱, 소형 모델 활용 검토 | API 비용 30% 이상 절감 |

### 8.2 품질 최적화

- **LLM 분류기 프롬프트 튜닝**: Few-shot 예제 추가, 분류 정확도 향상
- **엔터티 병합 (Entity Resolution)**: 동일 엔터티의 다양한 표현을 통합하여 그래프 품질 향상
- **HTML→Markdown 변환 개선**: 정규식 기반에서 markdownify 라이브러리로 전환하여 변환 품질 향상
- **청킹 전략 고도화**: 고정 토큰 단위에서 의미 단위(섹션, 문단) 기반 청킹으로 전환 검토

### 8.3 운영 최적화

- **설치 자동화**: 원클릭 설치 마법사 및 Docker 컨테이너화
- **모니터링**: 처리 이력 추적, 오류 로깅, 사용량 통계 대시보드
- **보안 강화**: OS Keyring 기반 자격증명 관리 (적용 완료), 접근 권한 관리 체계 추가 예정

### 8.4 확장성 최적화

- **데이터 소스 확장**: Jira, Slack, Google Drive 등 추가 연동 커넥터 개발
- **멀티 테넌트 지원**: 팀/조직 단위의 독립적 지식 공간 운영
- **벡터 DB 마이그레이션**: 현재 ChromaDB(로컬)에서 Milvus/Qdrant 등 분산 벡터 DB로 확장 가능
